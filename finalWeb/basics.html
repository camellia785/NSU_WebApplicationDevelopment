How Internet Works

The internet is a massive, interconnected network of devices‚Äîcomputers, phones, servers‚Äîcommunicating by sending small units of data called packets. Understanding how these packets travel from a user to a server (and back) is essential for backend developers.

1. What the Internet Is (Definition)

At a high level, the internet is built on three main pieces:

  • Clients ‚Äì devices like browsers, mobile apps, or API consumers that initiate requests.
  • Servers ‚Äì machines that process the request and return a response (HTML, JSON, files, etc.).
  • Routers & Network Infrastructure ‚Äì routers forward packets between networks, choosing the fastest and most reliable path.

Every request (like loading google.com) involves multiple routers forwarding packets until they reach the destination server.

2. How Data Moves (Packet Flow)

When you send a request:

  • Your device breaks the data into packets.
  • Each packet is labeled with source IP, destination IP, port numbers, and sequence numbers.
  • Packets travel independently through many routers.
  • The server receives all packets and reassembles them.
  • The server sends response packets back in the same way.

Even if packets take different routes, the server reconstructs them in order.

3. Why Backend Developers Should Care

Understanding internet flow helps you:

  • Debug slow or failing requests
  • Understand timeouts, retries, or packet loss
  • Communicate better with DevOps or Infra teams
  • Design scalable APIs and services
  • Interpret logs related to networking issues

A backend system does not run in isolation‚Äîevery action is part of a larger network.

4. Real Example

When a client calls:

GET https://api.example.com/users

This triggers:

  • DNS lookup
  • TCP connection (3-way handshake)
  • TLS negotiation (if HTTPS)
  • Request packet transmission
  • Response packets returning

Understanding each stage lets you quickly diagnose ‚Äúwhy something is slow or broken.‚Äù

HTTP / HTTPS

HTTP (HyperText Transfer Protocol) is the foundation of communication on the web. HTTPS is the secure, encrypted version of HTTP, protecting data in transit.

1. What HTTP Is (Definition)

HTTP is a stateless, request-response protocol used by browsers, mobile apps, and backend services.

A typical HTTP request includes:

  • Method ‚Äî GET, POST, PUT, DELETE
  • Headers ‚Äî metadata such as auth tokens, content type, user agent
  • Body ‚Äî data sent to the server (JSON, form data, etc.)
  • URL ‚Äî the resource being requested

A server responds with:

  • Status code (200, 404, 500‚Ä¶)
  • Response headers
  • Response body

2. Why HTTPS Exists

HTTP has no encryption, which means anyone on the network could read the data. To solve this, HTTPS combines HTTP with TLS/SSL (Transport Layer Security).

This adds:

  • Encryption ‚Äî prevents eavesdropping
  • Authentication ‚Äî ensures the server is legitimate
  • Integrity ‚Äî prevents data tampering

3. How HTTPS Works (TLS Handshake Overview)

  1. Client connects and says ‚ÄúI want to speak HTTPS.‚Äù
  2. Server sends its certificate (signed by a Certificate Authority).
  3. Client verifies the certificate's legitimacy.
  4. Both sides generate shared encryption keys.
  5. Encrypted communication begins.

This process happens in milliseconds but enables secure web browsing, APIs, payments, and login systems.

4. Role of Certificate Authorities (CAs)

CAs are trusted organizations (e.g., DigiCert, Google Trust Services) that issue digital certificates.

A certificate proves that:

  • The server is who it claims to be
  • The domain is valid
  • The connection must be encrypted

5. Why This Matters for Backend Developers

As a backend developer, understanding HTTP/HTTPS helps you:

  • Design APIs correctly
  • Handle headers, cookies, and authentication
  • Debug CORS and networking issues
  • Choose secure settings (TLS version, HSTS, etc.)
  • Understand why certain requests fail (expired certificate, wrong domain, insecure connection‚Ä¶)

6. Real Example

An HTTP request might look like:

GET /users HTTP/1.1
Host: api.example.com
Authorization: Bearer <token>
Accept: application/json

With HTTPS, all of this is sent inside an encrypted TLS tunnel.

Domain Name

A domain name is the human-readable address of a website or service ‚Äî for example, google.com instead of 142.250.72.14. Because humans cannot easily memorize long IP addresses, domains act as a friendly abstraction on top of the internet‚Äôs addressing system.

1. What a Domain Name Is (Definition)

A domain name is composed of hierarchical parts:

  • TLD (Top-Level Domain) ‚Äî .com, .org, .io
  • Second-level domain ‚Äî google, amazon, yeonadev
  • Subdomains ‚Äî www.google.com, api.example.com

Each domain maps to one or more IP addresses so that computers know where to find the correct server.

2. Why Domain Names Matter

  • Users would otherwise need to type IP addresses directly
  • Websites could not change servers/IPs without breaking links
  • Infrastructure becomes much easier to maintain

Domain names provide stability even when underlying servers change.

3. How Domains Are Connected to DNS

Domain names depend entirely on DNS to resolve names ‚Üí IP addresses.

Example:

api.myapp.com ‚Üí 203.0.113.16

DNS records define these mappings (A, AAAA, CNAME, MX, etc.).

4. Example Scenario

If your backend service moves to a new hosting provider:

  • You update the domain‚Äôs DNS records
  • Users keep using the same domain
  • Traffic automatically goes to the new server

5. Why Backend Developers Should Care

  • APIs often use subdomains like api.example.com
  • Microservices may use their own domains
  • Misconfigured DNS can break your backend instantly
  • SSL certificates require correct domain & DNS settings

Hosting & Browsers

Web hosting determines where your website or API physically lives, while browsers determine how users experience it. Together, they form the core of the user experience.

1. What Hosting Is (Definition)

Hosting is a service that stores:

  • Website files (HTML, CSS, JS)
  • Backend server code and APIs
  • Databases and static assets

Types of hosting include:

  • Shared Hosting
  • VPS (Virtual Private Server)
  • Cloud Hosting (AWS, GCP, Azure)
  • Serverless / Edge Hosting (Vercel, Cloudflare Workers)

2. How Browsers Work

Browsers perform multiple steps to render a webpage:

  1. Parse HTML
  2. Load CSS, JS, and images
  3. Build DOM tree
  4. Build CSSOM
  5. Combine into the Render Tree
  6. Paint the final layout to the screen

Browsers also send HTTP requests and interpret responses.

3. Why Hosting & Browsers Matter Together

Even if the server is fast, performance suffers if the browser:

  • Loads too many scripts
  • Uses render-blocking resources
  • Handles unoptimized images

4. Example Scenario

When loading myportfolio.com:

  • Browser sends request ‚Üí hosting server responds with HTML
  • Browser loads assets and renders UI
  • Scripts execute to enable interactions

5. Why Backend Developers Should Care

  • API latency affects page load speed
  • Backend response size impacts rendering time
  • Incorrect CORS headers break browser requests
  • Hosting misconfigurations lead to outages

DNS (Domain Name System)

DNS is the system that translates user-friendly domain names into IP addresses used by machines. It is often called the ‚Äúphonebook of the internet.‚Äù

1. What DNS Is

DNS is a distributed global database organized into several layers:

  • Root servers
  • TLD servers (.com, .net, .org)
  • Authoritative name servers
  • Recursive resolvers (ISP DNS, Google DNS)

2. How DNS Lookup Works

  1. Browser checks local DNS cache
  2. OS resolver checks cache
  3. Request is sent to a recursive resolver (e.g., 8.8.8.8)
  4. The resolver contacts the Root server
  5. Root refers to the TLD server
  6. TLD refers to the authoritative server
  7. Authoritative server responds with the IP address
  8. Browser finally sends HTTP/HTTPS request to that IP

3. Why DNS Matters

  • If DNS fails, your website becomes unreachable
  • DNS enables seamless server migration
  • Load balancers often use DNS routing
  • APIs and microservices depend on internal DNS

4. DNS Records Developers Should Know

  • A ‚Äî Domain ‚Üí IPv4
  • AAAA ‚Äî Domain ‚Üí IPv6
  • CNAME ‚Äî Domain alias
  • MX ‚Äî Mail servers
  • TXT ‚Äî Verification, SPF, DKIM
  • NS ‚Äî Name server delegation

5. Example Scenario

If your backend API moves from:

203.0.113.10 ‚Üí 203.0.113.20

You update DNS ‚Äî users still use the same domain, and traffic updates automatically.

6. Why Backend Developers Must Understand DNS

  • SSL certificates require DNS ownership
  • Microservice routing depends on DNS
  • API outages often stem from DNS issues
  • Incorrect DNS can break production instantly
